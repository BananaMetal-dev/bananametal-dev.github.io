# Banana Metal Apps & Music Site 実装仕様書

## 1. 目的

GitHub Pagesで公開する個人サイトを実装する。

このサイトは、以下の3つを主目的とする。

1. 作成したブラウザアプリを紹介し、利用ページへ誘導する
2. 作成した楽曲の一覧とYouTubeリンクを掲載する
3. Googleフォームへ誘導するお問い合わせ窓口を設置する

初期版は静的サイトとして公開する。
ユーザー登録、ログイン、決済、広告、アクセス解析、外部APIによる個人データ送信は実装しない。

---

## 2. サイトの基本方針

### 2.1 公開先

* GitHub Pages
* GitHubリポジトリから静的サイトとして公開する
* 初期URLはGitHub Pages標準URLを使う
* 独自ドメインは初期実装の対象外とする

### 2.2 想定サイト構造

```text
/
├─ Home
├─ Apps
├─ Music
├─ Contact
├─ Privacy
└─ 404
```

### 2.3 想定ユーザー

* 作成者の楽曲を聴きたい人
* 公開アプリを使いたい人
* 音楽生成、音楽制作、ビジュアライザー、プロンプト管理に関心がある人
* アプリの不具合報告、機能要望、連絡をしたい人

---

## 3. 初期版で実装するページ

### 3.1 トップページ `/`

表示内容：

* サイト名
* 簡潔な説明文
* Appsへの導線
* Musicへの導線
* Contactへの導線
* 注目アプリまたは注目楽曲の表示領域
* フッター

トップページは情報過多にしないこと。

最重要導線は以下の3つとする。

```text
Apps
Music
Contact
```

---

### 3.2 Appsページ `/apps/`

作成したアプリの一覧を表示する。

初期実装では、アプリ一覧の管理元はローカルの静的データファイルでよい。

各アプリカードには以下を表示する。

```text
- アプリ名
- 短い説明
- スクリーンショットまたはプレースホルダー画像
- 主な特徴
- 利用上の注意
- アプリを開くボタン
```

初期想定アプリ：

```text
1. 音楽ビジュアライザー
2. プロンプト管理アプリ
```

ただし、アプリ本体が未実装または未公開の場合でも、カード自体は表示できる構成にすること。

アプリカードには以下の状態を持たせる。

```text
- available
- coming_soon
- private
```

`private` のアプリは公開ページに表示しないこと。

---

### 3.3 Musicページ `/music/`

楽曲一覧をカード形式で表示する。

データソースは初期版では以下を使う。

```text
/data/songs.json
```

初期段階ではGoogleスプレッドシート連携を実装しない。
`songs.json`を手動で編集して動作確認できるようにする。

Musicページで表示する項目：

```text
- ジャケット画像
- 曲名
- アーティスト名
- 短い説明
- 公開日
- タグ
- YouTubeを開くボタン
```

表示条件：

```text
siteVisible === true の曲だけ表示する
sortOrder の昇順で表示する
featured === true の曲は注目曲エリアにも表示する
```

初期版ではYouTube iframe埋め込みを実装しない。
YouTubeを新しいタブで開くリンクボタンを実装する。

Musicページには、楽曲一覧の最終更新日時を表示する。

表示例：

```text
最終更新：2026年6月30日 00:07 JST
```

この日時は、ホームページ上の公開楽曲一覧が最後に変更された日時を意味する。

---

### 3.4 Contactページ `/contact/`

お問い合わせページを作る。

初期版では、フォーム送信機能をサイト内に実装しない。
Googleフォームへ遷移するボタンを設置する。

表示内容：

```text
- お問い合わせ案内
- 対象例
  - アプリの不具合
  - 機能要望
  - 楽曲について
  - 仕事・連携の相談
  - その他
- Googleフォームを開くボタン
- 返信には時間がかかる場合がある旨
```

GoogleフォームのURLは設定ファイルまたは環境に依存しない静的設定として管理し、後から容易に差し替えられるようにする。

GoogleフォームURLが未設定の場合は、ボタンを無効化し、準備中の表示を行うこと。

---

### 3.5 Privacyページ `/privacy/`

以下の内容を簡潔に掲載する。

```text
- お問い合わせ内容は、返信・対応のために利用する
- お問い合わせはGoogleフォームおよびGoogleスプレッドシートで管理される
- 公開アプリで音源・画像・動画を扱う場合、個別のアプリに記載された仕様に従う
- ブラウザ内処理型アプリでは、音源・画像・生成動画を当サイトへアップロードしない設計を目指す
- 外部サービスへのリンク先では、そのサービス側の規約・プライバシーポリシーが適用される
```

断定的に「一切の外部通信がない」と記載しないこと。
将来、公開プリセット取得や外部リンクが追加される可能性があるためである。

---

### 3.6 404ページ `/404.html`

表示内容：

```text
ページが見つかりません。
トップページへ戻る
Appsを見る
Musicを見る
```

GitHub Pages環境で動作すること。

## 4. 技術要件

### 4.1 推奨技術構成

以下を基本構成とする。

```text
- React
- TypeScript
- Vite
- CSS Modules または保守しやすい通常CSS
- 静的JSONデータ
```

不要なUIフレームワーク、UIキット、認証基盤、DB、バックエンドは追加しないこと。

### 4.2 GitHub Pages対応

GitHub Pagesで動く構成にする。

以下を満たすこと。

```text
- project site形式でもusername.github.io形式でもbase pathを変更しやすい
- GitHub Pagesの配下パスでアセット参照が壊れない
- リロード時に404にならないよう静的ページ構成または適切なフォールバックを採用する
- npm run build が成功する
```

### 4.3 スマホ対応

以下を満たすこと。

```text
- 横幅320pxから表示が崩れない
- 音楽カードはスマホでは1列表示
- ボタンは押しやすいサイズ
- 文字が極端に小さくならない
- 横スクロールが発生しない
```

---

## 5. デザイン方針

* 音楽とツールを扱う個人サイトとして、過度に企業サイト風にはしない
* 黒一色だけに寄せず、読みやすいコントラストを確保する
* 初期版では凝ったアニメーションを入れない
* ページ読み込み性能を優先する
* 楽曲カードとアプリカードは同じデザイン体系で統一する
* 画像がない場合は壊れず、プレースホルダーを出す
* YouTubeリンクは外部リンクであることが分かる表示にする

```text

- Googleサービスアカウント

- 外部APIキー
- 認証機能
- 会員登録
- クラウド保存
- 解析タグ
- 広告タグ
- Google Analytics
- 音源アップロード
- 画像アップロード
- 動画アップロード
- サーバーサイド処理
- YouTube iframe自動埋め込み
```

---

## 7. セキュリティ要件

公開リポジトリおよびGitHub Pagesに以下を含めないこと。

```text
- APIキー
- パスワード
- サービスアカウント認証JSON
- OAuthトークン
- Googleフォームの回答データ
- 非公開のYouTube URL
- 制作メモ
- Sunoプロンプト原文
- 個人情報
- メール認証情報
```

公開用データは、`data/songs.json`など必要最小限の静的データに限定する。

---

## 8. 初期版の完了条件

以下をすべて満たしたら初期版完了とする。

```text
1. トップページ、Apps、Music、Contact、Privacy、404が存在する
2. Appsページにアプリカードが表示される
3. Musicページがdata/songs.jsonを読み込んで曲カードを表示する
4. siteVisibleがfalseの曲は表示されない
5. sortOrder順で表示される
6. featured曲が注目曲エリアに表示される
7. YouTubeリンクが正しく生成される
8. ContactページからGoogleフォームへ遷移できる
9. GoogleフォームURL未設定時に壊れない
10. 最終更新日時がMusicページに表示される
11. スマホ幅320pxで横スクロールが発生しない
12. npm run buildが成功する
13. READMEに起動方法、ビルド方法、公開方法、songs.json編集方法が記載される
```
